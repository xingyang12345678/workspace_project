# AI 数据工作台架构文档

本文档以**清晰的结构和通俗的语言**，帮助初学者理解整个项目的架构。

---

## 一、整体结构

这是一个典型的 **前后端分离** 应用，包含两个独立部分：

```
AI 数据工作台/
├── backend/          ← 后端（FastAPI + Python）
│   ├── main.py       ← 程序入口：注册所有 API 接口
│   ├── api/          ← 各种功能的 API 接口（HTTP 路由）
│   ├── services/     ← 业务逻辑：处理数据、调用模型等
│   ├── models/       ← 数据模型：定义请求/响应的数据结构
│   ├── core/         ← 核心工具：异常、类型定义等
│   └── services/
│       ├── file_service.py      ← 文件系统操作（读写、备份）
│       ├── dataset_service.py   ← JSONL 数据读取与解析
│       ├── token_service.py     ← Token 统计（调用 HuggingFace）
│       ├── chat_stats_service.py ← N-gram、字符串查找统计
│       ├── knowledge_service.py ← 知识库管理
│       ├── script_service.py    ← 用户脚本运行
│       └── ...
│
├── frontend/         ← 前端（React + Vite）
│   ├── src/
│   │   ├── App.tsx           ← 程序入口：定义页面路由
│   │   ├── components/       ← 可复用组件（Layout、Terminal等）
│   │   ├── features/         ← 页面功能模块（DatasetViewer、DocsCenter等）
│   │   ├── api/              ← API 调用封装（前端调用后端的接口）
│   │   └── ...
│   └── ...
│
└── docs/             ← 文档（你正在看的这个目录）
```

### 关键概念

| 概念 | 说明 |
|------|------|
| **后端** | 提供数据与计算服务，用 HTTP 接口给前端调用 |
| **前端** | 用户界面，展示数据、接收用户操作、调用后端接口 |
| **API 接口** | 后端暴露的 URL（如 `/api/datasets/token-count`），前端通过 HTTP 请求调用 |
| **Workspace** | 本地工作区，存放用户数据（datas、knowledge、scripts等） |

## 二、数据流程（从用户点击到显示）

以「查看 JSONL 并统计 Token」为例：

1. **用户操作**  
   打开「数据阅读」→ 点击左侧 `datas/sample.jsonl` → 点击「统计全文件」。

2. **前端处理**  
   - `App.tsx` 根据路由 `/datasets?path=xxx&file=yyy` 渲染 `DatasetViewer.tsx`。
   - `DatasetViewer` 调用 `getRecords(...)` → 从 `/api/datasets/records` 获取数据。
   - 用户点击「统计全文件」按钮 → 调用 `tokenStats(...)` → 向后端 `/api/datasets/token-stats` 发 POST 请求。

3. **后端处理**  
   - `main.py` 接收请求，路由到 `datasets.py` 的 `post_token_stats` 函数。
   - `post_token_stats` 调用 `token_service.token_stats_file(...)`。
   - `token_stats_file`：
     - 读取 `datas/sample.jsonl` 的所有记录（`dataset_service.get_all_records`）。
     - 对每条记录计算 chosen/rejected 的 token 数（调用 `transformers` 模型）。
     - 统计均值、标准差、分布直方图等。

4. **结果返回与渲染**  
   - 后端返回统计结果 JSON → 前端 `tokenStats(...).then(...)` 接收 → 存入 `setTokenStatsResult`。
   - `DatasetViewer` 的 UI 根据 `tokenStatsResult` 渲染：  
     - 显示均值、标准差、P25/P75/P90、直方图。  
     - 同时自动请求当前条的 token → 更新气泡显示。

### 图解流程

```
┌─────────────────┐
│    用户操作     │
│ (点击按钮)      │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│   前端 DatasetViewer  │

└────────┬────────┘
```

## 三、核心模块详解

### 1. 后端模块

#### 1.1 主入口 `main.py`

```python
app = FastAPI()
app.include_router(datasets.router)   # 注册数据集接口
app.include_router(knowledge.router)  # 注册知识库接口
# ... 更多 router
```

**作用**：启动 FastAPI 服务，注册所有 API 路由（`/api/datasets/*`, `/api/knowledge/*` 等）。

---

#### 1.2 API 层 `api/`

每个子模块对应一个功能领域：

| 文件 | 功能 | 举例接口 |
|------|------|----------|
| `datasets.py` | 数据集相关 | `/token-count`, `/ngram`, `/string-search`, `/run-script` |
| `knowledge.py` | 知识库 | `/create`, `/update`, `/delete`, `/archive` |
| `tools.py` | 工具执行 | `/list`, `/run` |
| `backup.py` | 备份 | `/datas` (POST) |
| `scripts.py` | 脚本管理 | `/list`, `/save`, `/delete` |

**特点**：
- 使用 Pydantic 定义请求体（`Request` 类）和响应体。
- 处理异常，返回 HTTP 状态码（200/400/404/503）。

---

#### 1.3 业务服务层 `services/`

业务逻辑所在，**不直接处理 HTTP**，只处理纯数据操作：

| 文件 | 功能 |
|------|------|
| `dataset_service.py` | 读取 `datas/` 下 JSONL，提取 messages/chosen/rejected 文本 |
| `token_service.py` | 加载 HuggingFace tokenizer，计算 token 数，缓存结果 |
| `chat_stats_service.py` | N-gram 统计、字符串查找与计数 |
| `knowledge_service.py` | 读写 `knowledge/entries.jsonl` |
| `script_service.py` | 执行用户脚本，注入 `workspace` 对象 |
| `file_service.py` | 文件读写、备份、删除（带安全校验） |

**设计原则**：
- 业务逻辑与接口解耦：`services/` 可被测试、复用，不依赖 FastAPI。
- 所有函数都是**纯函数**：输入参数 → 返回结果，不直接读写全局状态（缓存除外）。

---

#### 1.4 数据模型 `models/`

定义接口的「请求体」和「响应体」结构：

```python
# 请求（前端传给后端）
class TokenStatsRequest(BaseModel):
    path: str
    file: str
    model: str
    scope: str = "both"

# 响应（后端返回给前端）
class TokenStatsResult(BaseModel):
    mean: float
    min: int
    max: int
    n: int
    std: float
    median: float
    p25: float
    p75: float
    p90: float
    histogram: {"bucket_edges": [], "counts": []}
```

**作用**：
- 明确接口的「契约」：前端知道要传什么、后端知道要返回什么。
- Pydantic 自动做数据校验与转换。

---

### 2. 前端模块

#### 2.1 主入口 `App.tsx`

```tsx
<BrowserRouter>
  <Routes>
    <Route element={<Layout />}>
      <Route path="/" element={<ModuleHub />} />
      <Route path="/datasets" element={<DatasetViewer />} />
      <Route path="/docs" element={<DocsCenter />} />
      <Route path="/system" element={<SystemPage />} />
      {/* ... */}
    </Route>
  </Routes>
</BrowserRouter>
```

**作用**：
- 使用 React Router 定义「路由」：URL → 对应的组件。
- `Layout` 是全局布局（导航栏、头部、终端），嵌套的 `Outlet` 渲染各功能页。

---

#### 2.2 功能模块 `features/`

每个 `.tsx` 对应一个功能页面：

| 文件 | 功能 |
|------|------|
| `DatasetViewer.tsx` | 数据阅读：气泡展示、Token 统计、N-gram、字符串查找、代码 |
| `DocsCenter.tsx` | 文档阅读：可拖动窗口、Tab 切换（人读版 / AI 版 / 接口文档） |
| `KnowledgeManager.tsx` | 知识库管理：查看、编辑、按天归档 |
| `SystemPage.tsx` | 系统管理：清除缓存、导出/清空数据 |
| `FileExplorer.tsx` | 左侧文件树（导航） |
| `TerminalPanel.tsx` | 底部终端面板 |

**特点**：
- 每个组件都是 **自包含的**：状态、逻辑、渲染都在一个文件中（React Hooks）。
- 使用 `useState` 管理 UI 状态（当前页、加载中、结果）。

---

#### 2.3 API 调用层 `api/`

封装对后端 HTTP 的调用，**避免在组件里直接写 `fetch`**：

```typescript
// api/datasets.ts
export function tokenCount(path: string, file: string, index: number, model: string) {
  return post<TokenCountResult>("/api/datasets/token-count", { path, file, index, model });
}

export function tokenStats(path: string, file: string, model: string, scope: string) {
  return post<TokenStatsResult>("/api/datasets/token-stats", { path, file, model, scope });
}
```

**作用**：
- 集中管理所有接口调用。
- 统一处理错误、序列化。

## 四、Workspace 目录结构

Workspace 是**用户数据的根目录**（默认 `workspace_project/`）：

```
workspace_project/
├── datas/              ← 数据集（你自己的 JSONL）
│   ├── chat.jsonl
│   └── sft/
│       └── data.jsonl
├── knowledge/          ← 知识记录
│   └── entries.jsonl
├── tools/              ← Python 脚本
│   ├── summarizer.py
│   └── cleaner.py
├── pipelines/          ← Pipeline 定义
│   └── my_pipeline.json
├── backup/             ← 备份（自动管理，无需操作）
│   └── manifest.json
└── .ai-workspace/      ← 用户自定义脚本（数据集分析用）
    └── dataset_scripts/
        └── analyze.py
```

**特点**：
- 所有数据都在本地，**无需云端存储**。
- 可通过 `WORKSPACE_ROOT` 环境变量自定义路径。
- 清除缓存时**不会删除**这些文件，只删内存/缓存/`__pycache__`。

## 五、技术栈与依赖

### 后端（Python）

| 库 | 用途 |
|----|------|

## 六、启动与开发

### 一键启动（推荐）

在项目根目录：

```bash
npm install          # 首次运行需安装依赖
npm start            # 同时启动后端 (8000) + 前端 (5173)
```

或使用脚本：

```bash
chmod +x start.sh
./start.sh
```

### 分别启动

**后端**

```bash
cd backend
pip install -r requirements.txt
uvicorn main:app --reload --host 0.0.0.0 --port 8000
```

**前端**

```bash
cd frontend
npm install
npm run dev
```

访问：`http://localhost:5173`

## 七、如何扩展

### 1. 新增 API 接口

**步骤**：
1. 在 `backend/models/dataset.py` 定义 `Request`/`Response`。
2. 在 `backend/services/` 新增业务逻辑函数。
3. 在 `backend/api/datasets.py` 新增 `@router.post("/xxx")`。
4. 在 `frontend/src/api/datasets.ts` 封装调用函数。
5. 在前端组件调用该 API。

### 2. 新增页面功能

1. 在 `frontend/src/features/` 新建 `.tsx`。
2. 在 `App.tsx` 添加路由 `<Route path="/xxx" element={<NewPage />} />`。
3. 在 `ModuleHub/modules.ts` 添加模块配置。

## 八、调试与排错

### 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| 数据不显示 | 后端未启动 / 未读到 JSONL | 访问 `http://127.0.0.1:8000/docs` 检查后端；检查 `datas/` 下文件 |
| Token 统计失败 | transformers 未安装 / 模型下载失败 | `pip install transformers`；检查网络/代理 |
| 前端访问后端失败 | 端口不对 / CORS | 检查后端运行端口；确认 Vite 代理配置 |

### 调试工具

- **后端**：查看终端日志；`http://127.0.0.1:8000/docs` 的 Swagger UI。
- **前端**：浏览器 F12 → Console 查看错误；Network 标签看 API 请求。

## 九、总结

| 层级 | 组件 | 作用 |
|------|------|------|
| **用户交互层** | `App.tsx`, `features/*.tsx` | 渲染 UI，接收用户操作 |
| **API 层** | `api/`, `backend/api/*.py` | 封装 HTTP 请求与路由 |
| **业务逻辑层** | `services/` | 处理数据、调用模型 |
| **数据层** | `models/`, `workspace/` | 定义数据结构，存放用户数据 |
| **基础设施** | `main.py`, `vite.config.ts` | 启动服务、代理转发 |

整个项目的核心设计原则：

1. **分层清晰**：UI、API、业务、数据各司其职，便于维护。
2. **本地优先**：所有数据在本地 workspace，隐私安全。
3. **可扩展**：新增功能只需添加对应模块，无需改动核心。
4. **易用性**：一键启动、清缓存、导出/清空数据，降低用户门槛。
